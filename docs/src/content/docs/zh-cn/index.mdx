<div class="dux-section">
  <div class="dux-section-head">
    <span class="eyebrow">为什么要再做一个 ORM</span>
    <h2>取其精华，去其繁文缛节</h2>
    <p>主流 ORM 各有所长。Dux 保留这些优点，去掉 Go 开发者真正会感到的摩擦。</p>
  </div>

  <div class="dux-why">
    <article>
      <h3>比 GORM 更克制</h3>
      <p>GORM 功能很全、久经考验，但历史包袱让零值、tag 和隐式行为容易混乱。Dux 把 API 面积保持得足够小，能装进脑子里。</p>
    </article>
    <article>
      <h3>像 Eloquent 一样可读</h3>
      <p>Laravel Eloquent 读起来很优雅。Dux 用泛型保留这份流畅 —— 又不必为每次早期结构调整都写一个迁移文件。</p>
    </article>
    <article>
      <h3>没有生成的 client</h3>
      <p>很多 Node ORM 依赖生成的 client 或大量运行时 <code>any</code>。Dux 通过 Go 泛型做到类型安全，没有任何需要重新生成的东西。</p>
    </article>
  </div>
</div>

<div class="dux-section">
  <div class="dux-section-head">
    <span class="eyebrow">一屏看懂</span>
    <h2>定义、同步、查询</h2>
    <p>完整闭环 —— 模型、结构同步、类型化 CRUD —— 全程不离开 Go，也不用运行任何工具。</p>
  </div>

```go
type Product struct {
    oro.Model
    Code  string
    Price uint
}

func (Product) Define(s *oro.SchemaBuilder) {
    s.Table("products")
    s.Field("Code").String().Unique()
    s.Field("Price").Uint().Default(0)
}

// 根据上面的定义注册并同步结构
db.Register(Product{})
db.Sync(ctx)

// 类型安全、可读的 CRUD，使用语义化条件
product, _ := db.Use[Product]().Create(ctx, &Product{Code: "P001", Price: 100})
found, _ := db.Use[Product]().Where(oro.Field("Code").Eq("P001")).First(ctx)
```
